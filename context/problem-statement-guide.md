# Problem Statement Guide

> This file defines the framework Claude uses to rewrite raw stakeholder feedback into structured problem statements.

## Framework: Jobs To Be Done (JTBD)

A good problem statement answers:
- **Who** is experiencing the problem?
- **What** are they trying to accomplish (the job)?
- **When** does the problem occur?
- **Why** does the current situation fall short?
- **What is the impact** of not solving it?

## Template

> When [situation], [user type] struggles to [accomplish goal] because [root cause/barrier], which results in [negative impact].

## Rules
- Do NOT include solutions in the problem statement
- Focus on the underlying need, not the feature request
- Use plain language — avoid internal jargon
- Be specific: avoid vague words like "better", "faster", "easier" without context

## Example

**Raw feedback:** "We need a dashboard for managers."

**Problem statement:** When preparing for weekly team meetings, managers struggle to get a consolidated view of their team's performance because data is spread across three different tools, which results in 30–60 minutes of manual work before each meeting.
